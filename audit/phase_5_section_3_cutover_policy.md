Phase 5 §3 — Layer 2 Cutover Policy
Status: Authoritative source for Layer 2 manifest drafting.
Scope: §3 only. Phase 5 §§1–2 (strategic synthesis, layer-by-layer plan) not yet written; this section is extracted ahead of them because Layer 2 prep requires it.
Authored: 2026-04-23. Inputs: Step 1 baseline inventory, phase 1a/1b/2a/2b verification records, phase 4 data integrity findings, live operator context (MS Vallee, owner-operator test site, v1 Flutter dormant 30+ days).

3.0 Principle
Layer 2 cutover partitions every row in the live Supabase instance into exactly one of three sets:

Preservation set — configured state that encodes expensive-to-recapture human or operational intent. Preserved bit-for-bit across cutover.
Wipe set — observed state (the event corpus) produced by a test-grade site with no downstream consumers. Truncated.
Untouched set — tables that are empty, unused, or whose contents are regenerated automatically on next boot/poll. No action taken; cutover neither preserves nor wipes.

The partition is exhaustive. Every table in the schema must map to exactly one set. Any table CC cannot confidently assign to a set during manifest drafting is a policy gap in this document and must be escalated back for a §3.1/§3.2 amendment — not resolved by CC judgement.
The preservation principle is configured state, not observed state. A row is preserved if recapturing it post-wipe would require human action (re-enrollment, re-provisioning, re-pairing, re-configuration). A row is wiped if it was generated by the system observing the world and has no enduring operational meaning.

3.1 Preservation set
The preservation set is closed. Seven categories. Nothing outside these categories is preserved.
3.1.1 Face gallery cluster
Enrolled human identity records for MS Vallee residents (five enrolled as of 2026-04). Includes identity records, face embeddings/vectors (whether in the same table or a separate embeddings table), and any enrollment audit trail (who was enrolled, when, by whom).
Rationale: re-enrollment requires physical presence of the enrollee and fresh capture. Non-trivial friction.
3.1.2 Vehicle registry cluster
Registered vehicles for MS Vallee (four as of 2026-04). Includes vehicle records, owner linkage, plate data, and any alert-suppression rules tied to enrolled vehicles.
Rationale: suppression rules encode operator intent ("don't alert on my own car"). Losing them produces alert floods on the next site visit.
3.1.3 Zone definitions cluster
Geometric and logical zones defined on the MS Vallee site. Includes zone records and zone-to-camera mappings where they exist. Zone-to-alert-rule mappings are not preserved because they do not yet exist in the schema (to be confirmed by CC against live schema during Phase A).
Rationale: zones encode site topology. Re-drawing zones requires operator time on the dashboard with the site map open.
3.1.4 Site configuration cluster
The MS Vallee site record and all rows that hang off it encoding operational state: operating hours, contact roster, escalation rules, armed response linkage, any site-level settings or feature flags.
Rationale: site config is the spine the entire deployment is assembled around. Losing it means reconstructing deployment state from memory.
3.1.5 Camera inventory cluster
Camera records plus their dependent configured state: RTSP credentials, ISAPI credentials, channel mappings, AI model assignments, per-camera detection thresholds or suppression windows.
Rationale: camera credentials are site-specific and stored in the platform because they're not stored anywhere else authoritative. Wiping them means re-entering credentials for every channel.
3.1.6 Telegram bridge configuration
Chat-ID to site mappings, operator-to-Telegram identity links, bot-level configuration rows. The Telegram message/update history (telegram_inbound_updates etc.) is not part of this cluster — that is event corpus, wiped per §3.2.
Rationale: re-pairing Telegram for every operator requires each operator to re-execute /start from their phone. Avoidable friction.
3.1.7 Edge device registration
Rows representing provisioned edge devices. Currently one: the Raspberry Pi 4B at 192.168.0.67 (hostname onyx-pi-msvallee). Includes the device record itself and any provisioning-side configuration (not heartbeat/health telemetry, which is event corpus).
Rationale: the Pi holds real provisioning state tied to a specific device and site. Re-registration is friction and introduces risk of misconfiguration.
3.1.8 Auth — special case
auth.users and related Supabase auth schema tables are preserved as-is. No wipe, no addition. Current contents are test logins; Layer 2 is not the moment to re-provision auth. Real operator accounts will be created during commercial onboarding in a later layer.
This is called out separately from the seven clusters above because auth.* is managed by Supabase and the preservation mechanism is "do not touch," not "export and restore."

3.2 Wipe set
The wipe set is the event corpus: every row generated by the system observing the world or by test-grade operator activity at MS Vallee that has no enduring operational meaning.
Explicitly wiped, non-exhaustive enumeration (CC resolves to table names during Phase A against live schema):

Incidents, alarms, alerts (all priority/status tiers)
Detection events (YOLO inference results, face recognition results, motion events)
Triage notes, operator dispositions, acknowledgement trails
Dispatch records, guard task assignments, patrol logs
Audit logs, activity logs, system event logs
Telegram message/update history (inbound polls, outbound sends, delivery receipts)
Edge device heartbeat / health telemetry
Any ingestion buffer, queue, or staging table populated by the camera worker, Telegram processor, or edge functions

The test for wipe-set membership: if the row was written by automated ingestion or by operator interaction during test-grade use, and recapturing it has no business value, it is wiped.
Scope note: the 30-day QA corpus freeze produced by scripts/cutover_qa_corpus_freeze.py exports a subset of the wipe set to JSON before truncation for post-cutover QA reference. This is not a preservation exception — the exported JSON is an archival artifact outside the live schema, not rows restored after wipe.

3.3 Untouched set
Tables that cutover does not preserve and does not wipe because either:

The table is currently empty and has no rows to act on (e.g., zone_alert_rules if confirmed empty during Phase A), or
The table's contents regenerate automatically on next boot, next poll, or next scheduled job (e.g., device heartbeat tables where the first heartbeat after cutover re-populates the row), or
The table is Supabase-managed infrastructure (storage.*, realtime.*, supabase_migrations.*, vault.*, etc.) and cutover must not touch it.

Untouched-set membership is the default CC assigns when a table is neither in §3.1's closed list nor matches §3.2's wipe criteria. Any table CC cannot confidently place in one of the three sets is flagged for §3 amendment rather than silently defaulted.

3.4 Execution sequence
Applied during Layer 2 execution (not prep). Prep produces the tooling that performs steps 4–7 when executed.

Pre-flight verification. Drift detector green. All Phase A manifest ambiguities resolved. 4b constraint migrations staged at supabase/manual/post_cutover_constraints/ and reviewed.
Operator confirmation gate. Explicit --yes or equivalent affirmative from operator in chat. No automation past this point without operator-present execution.
QA corpus freeze. scripts/cutover_qa_corpus_freeze.py exports last 30 days of wipe-set rows to dated JSON archive. Verified non-empty and parseable before proceeding.
Preservation export. scripts/cutover_preservation_export.py exports the §3.1 preservation set to JSON as belt-and-braces backup. Verified row counts match live schema counts before proceeding.
Wipe. Truncate wipe-set tables in FK-dependency order per manifest.yaml. RESTART IDENTITY CASCADE where applicable. Preservation and untouched sets not touched.
Post-wipe verification. Preservation-set row counts re-verified against step 4 export. Any mismatch aborts and triggers restore from step 4 JSON.
4b constraint application. Apply staged supabase/manual/post_cutover_constraints/ migrations. These constraints could not land in 4a because existing wipe-set rows would have violated them; post-wipe, the tables are empty or compliant.
Drift detector re-run. Must return green before cutover is declared complete.
Forward discipline. All further schema changes go through supabase/migrations/. No out-of-band ALTERs. No exceptions. This is the Layer 1 discipline Layer 2 inherits and does not relax.

Between steps 5 and 7 the live schema is in a transitional state: wipe-set tables are empty, 4b constraints not yet applied. Duration should be minutes, not hours. No operator actions against the dashboard during this window.

3.5 Scope boundary
This document defines cutover policy for MS Vallee, the only live site at Layer 2. When a second site is onboarded (commercial, Layer 5+), this policy does not automatically apply — commercial sites are never wiped, and their preservation set is the entire operational dataset. This document is a one-time test-site reset policy, not a general operational procedure.
Any future reference to "cutover" in the ONYX platform should not cite this document as precedent for wiping commercial data.
