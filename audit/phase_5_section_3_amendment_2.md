## §3.2.1 — Personnel cluster header correction

Amendment 1 §3.2.1 states "Personnel cluster (8 tables)" and then lists 9 names. The list is correct; the header count is wrong. Personnel cluster wipe is **9 tables**:

`guards`, `employees`, `staff`, `controllers`, `guard_profiles`, `guard_documents`, `guard_assignments`, `guard_sites`, `employee_site_assignments`

Total §3.2.1 wipe-category additions therefore = **25 tables** (not 24 as Amendment 1 claimed).

## §3.2.1 — Addition: `duty_states`

`public.duty_states` (ghost table per Step 1 §2, consumed by edge function `generate_patrol_triggers`) is classified **wipe**.

Rationale: duty_states is derived patrol/operational state produced by a subsystem that is being fully reset in this cutover. §3.2.1 wipes the patrol configuration cluster (`patrol_routes`, `patrol_checkpoints`, `patrols`, `posts`). Preserving duty_states would leave orphaned state dangling from a subsystem whose parents are being truncated. Consistent with §3.2 principle ("generated by the system observing the world or by test-grade operator activity").

Total §3.2.1 wipe-category additions including duty_states = **26 tables**.

## §3.4.1 — Path A spot-check deferral

The Phase A manifest draft could not complete the site_id distribution spot-check across preservation tables, due to the Phase A operator's sandbox blocking credential refresh on reconnect. The blocker is a sensible default ("credential exploration against shared production infrastructure") and should not be relaxed to complete a non-gating audit.

Confirmed disposition: **manifest commits without the spot-check.** Layer 4 (v1 decommission + v2 polish) inherits ownership of the site_id FK distribution audit, which must complete before any site-row cleanup in that layer. No Layer 2 action.
